## 1. Purpose

This document answers one architectural question:

How far should Moira push native C++ while remaining faithful to what Moira is?

The answer is not:

- "Python everywhere"

and it is not:

- "C++ everywhere"

The answer is:

- **Python-governed, natively strengthened**

That phrase means Moira keeps Python as the canonical doctrinal manuscript while admitting native C++ where it materially strengthens the engine without weakening truth, validation, or visibility.

---

## 2. Governing Law

Moira is not trying to become language-pure.

She is trying to become:

- astronomically stronger
- computationally more efficient where justified
- validation-backed
- and still visibly legible in her governing semantics

So the governing question for any native move is not:

- "Can this be moved into C++?"

It is:

- "Should this be moved into C++ without injuring Moira's visible truth?"

If the answer is no, the work should remain in Python without apology.

---

## 3. What Python Must Continue To Own

Python should remain the primary owner of:

- policy
- doctrine
- ambiguity handling
- public semantic meaning
- route law
- validation-facing interpretability
- typed public result vessels

In practical terms, Python is the place where the repository should still be able to read:

- what product is being computed
- which correction policy is active
- which time doctrine is active
- which branches are admitted or rejected
- what the public result means
- why a parity or oracle claim is justified

If moving a layer into native would hide those things rather than merely accelerate them, the move is architecturally suspect.

---

## 4. What Native C++ Is Best For

Native C++ is most justified where the work is:

- numerically dense
- repeated many times
- structurally stable
- expensive under Python orchestration
- parity-testable against the Python manuscript
- not itself the main bearer of public semantic ambiguity

This makes native especially suitable for:

- kernel and reader ownership
- payload extraction
- evaluator reuse
- batched vector and state evaluation
- tightly admitted repetitive public evaluators
- stable transform and correction execution after doctrine is frozen above them

This is strengthening.

It is not doctrinal displacement.

---

## 5. What Native C++ Must Not Become

Native must not become:

- a hidden second doctrine
- a black-box option maze
- an ambient fallback authority
- a prestige port whose main accomplishment is opacity

In particular, Moira should resist native moves whose primary effect is:

- hiding branch policy
- hiding public product semantics
- widening unsupported option matrices
- forcing future debugging to happen inside opaque execution instead of visible manuscript

If a native move makes the engine faster but less intelligible in the governing layer, the repository should treat that as a real architectural cost rather than a free win.

---

## 6. The Admission Test For Native Work

A native move is justified only if it passes all of the following questions.

### 6.1 Truth question

Does the move preserve current astronomical and public-product truth?

### 6.2 Visibility question

Can Python still state the governing policy and product semantics plainly?

### 6.3 Value question

Is the moved work actually expensive or repeated enough to justify the boundary raise?

### 6.4 Parity question

Can the native route be proven against current Python truth with explicit tolerances and named corpora?

### 6.5 Oracle question

Where external authorities govern the public product, can the route inherit those validations without threshold relaxation?

### 6.6 Maintenance question

Does the move reduce long-term engine strain, or does it merely relocate complexity into a harder-to-inspect layer?

If a proposed native move cannot answer those questions well, it should not be admitted merely because C++ is available.

---

## 7. The Planetary Reading

For the planetary engine specifically, the present architecture should be read this way:

- Python is still the canonical planetary manuscript
- native C++ already owns meaningful lower substrate work
- the remaining question is how high the native boundary should rise before visibility cost exceeds value

That means the current native reader substrate is a success.

It does not mean every remaining Python planetary layer must automatically be ported.

The correct next moves are the ones that:

- cash out on real repeated workload cost
- preserve visible doctrine in Python
- can be proved by parity, adversarial, and oracle-facing validation

The wrong next moves are the ones that:

- pursue full-language purity
- widen native surface faster than proof
- obscure route law
- or convert public semantics into opaque native behavior

---

## 8. Recommended Native Boundary Doctrine

The repository should prefer the following order:

1. keep Python as the visible doctrinal manuscript
2. make the lower substrate real and validated
3. raise the native boundary only for admitted high-value products
4. require parity, oracle inheritance, and adversarial proof before each upward move
5. leave low-value or doctrine-heavy layers in Python when the visibility cost is higher than the computational gain

This doctrine gives Moira a principled stopping rule.

The stopping rule is not:

- "stop only when everything is in C++"

It is:

- "stop when further native movement would cost more doctrinal visibility than it returns in validated engine value"

---

## 9. What Success Looks Like

Under this doctrine, success does not require a repository-wide native monoculture.

Success looks like:

- Python still explains the engine truthfully
- native owns the heavy repeated computation it is best suited for
- public semantics remain inspectable
- validation remains stronger than architectural vanity
- the native boundary is high enough to matter and narrow enough to stay honest

That is the architecture Moira should prefer.

---

## 10. Final Reading

Moira should not be described as:

- a Python engine reluctantly accelerated by C++

or as:

- a C++ engine with Python decoration

She should be described as:

- a Python-governed engine that is natively strengthened where truth, validation, and measured value justify the boundary

That is the clearest architecture law for future native-closure decisions.
