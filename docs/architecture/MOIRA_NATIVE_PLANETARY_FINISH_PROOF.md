## 1. Purpose

This document answers one narrow question:

What would it take to truthfully say that the entire native planetary engine is finished?

This is not a roadmap.

It is not a performance pitch.

It is a finish standard.

It exists so the repository does not confuse:

- a real native substrate
- a partially native planetary path
- and a fully certified native planetary engine

---

## 2. Current Truth

The repository is not there yet.

The governing current-state reading remains:

- native is materially real below the reader boundary
- Python still owns most of the canonical planetary manuscript above that boundary

That means the following are currently native-active or materially native-backed:

- DAF/SPK catalog reading
- supported segment payload extraction
- supported Chebyshev and type-13 evaluation inside the reader layer
- supplemental small-body reader ownership for admitted kernels

It also means the following remain Python-owned in the canonical path:

- barycentric route chaining
- Earth/geocentric state construction
- apparent correction orchestration
- coordinate transforms into public planetary products
- public result packaging

So the honest present-tense reading is not:

- "the native planetary engine is finished"

It is:

- "the native reader substrate is real"
- "the canonical planetary path is partially native"

---

## 3. What "Finished" Must Mean

The repository must not use the word `finished` loosely.

For this document, the entire native planetary engine is finished only when all of the following are true:

1. the canonical public planetary products are routed through admitted native execution for their core computational path
2. the native route preserves current Python public doctrine exactly in meaning
3. the current external validation inheritance remains green without relaxed thresholds
4. the native route survives adversarial and oracle-hostile testing on the public products, not only on substrate helpers
5. benchmark artifacts show stable, honest value on the real public workloads
6. documentation states the boundary plainly and no longer relies on ambiguous "partially native" language for the claimed closed products

If any of those remain open, the engine is not finished.

---

## 4. Surfaces That Must Be Accounted For

No full-finish claim is lawful unless the repository names the exact public surfaces covered by that claim.

At minimum, the finish claim must account for:

- `planet_at(...)`
- `all_planets_at(...)`
- `sky_position_at(...)`
- `heliocentric_planet_at(...)`
- `planet_relative_to(...)`
- the chart-facing facade routes that depend on them

Each of those surfaces must be classified explicitly as one of:

- `native closed`
- `intentional Python`
- `out of closure claim`

If a surface remains canonical, materially used, and still depends on Python for its core computation, it cannot be quietly omitted from an "entire engine" claim.

## 5. Boundary-Raising Requirement

The present native boundary is too low for a full-engine claim.

The engine cannot be called finished while native ownership stops at the reader layer.

For a full finish claim, the repository must either close or intentionally retain in Python each of the following layers:

### 5.1 Vector construction layers

- barycentric route chaining
- Earth barycentric state construction
- geocentric subtraction
- relative-vector assembly

### 5.2 Apparent doctrine layers

- light-time iteration
- annual aberration
- gravitational deflection
- frame bias / precession / nutation composition
- topocentric parallax where relevant
- diurnal aberration where relevant
- refraction where relevant

### 5.3 Coordinate and product layers

- ecliptic projection
- equatorial conversion
- horizontal conversion
- public speed derivation where exposed
- public planetary payload assembly needed to build `PlanetData` and `SkyPosition`

The finish claim does not require Python-facing result vessels to move into C++.

It does require that the core planetary computation no longer depends on Python-owned execution for the claimed native-closed products.

---

## 6. Doctrine Preservation Requirement

Native closure is not permission to invent a second planetary doctrine.

For every admitted native product, the repository must preserve:

- current public time-input meaning
- current Delta T policy meaning
- current apparent versus geometric doctrine
- current center and frame semantics
- current `PlanetData` and `SkyPosition` field meanings
- current error behavior for unsupported and disallowed modes
- current sign, sign-degree, and retrograde semantics

If native changes execution location but not meaning, that is lawful.

If native changes meaning, even subtly, the product is not finished. It is forked.

---

## 7. Python-Parity Proof Requirement

Before a product can be called native-finished, native must prove exact public parity to current Python doctrine on a fixed corpus.

This proof must include:

- all canonical chart bodies
- modern dates
- historical dates where the public route admits them
- future-policy dates
- split-JD cases where the reader and time path expose them
- exact boundary cases
- near-boundary continuity neighborhoods

The parity rule is strict:

- residuals should be attributable only to floating-point execution-order differences
- any systematic body-shaped, epoch-shaped, or mode-shaped residual is a failure until explained

This is stricter than external-oracle tolerance because native is not implementing a different doctrine.

It is implementing the same doctrine in a different execution location.

---

## 8. External Validation Inheritance Requirement

Python parity alone is not enough.

A finished native planetary engine must inherit the current doctrine-facing validation surfaces without relaxed thresholds.

That includes, where applicable:

- Horizons-backed apparent planetary validations
- continuity and stress slices already used to guard the public planetary path
- topocentric public-product validations
- small-body supplemental validations for any admitted native-backed public route

This protects against the most dangerous false finish:

- native matches Python
- Python and native drift together
- the repository mistakes internal agreement for truth

---

## 9. Adversarial Requirement

A finished engine must survive hostile testing above the substrate.

It is not enough for the native DAF reader to be adversarially validated.

The planetary public routes themselves need hostile proof on the native path.

At minimum, the finish corpus should include:

- seam continuity across supported SPK segment boundaries on public products
- wrap-boundary and time-boundary continuity on public planetary outputs
- future-policy stability on admitted future cases
- topocentric continuity where sky products are admitted
- route-equivalence checks across ecliptic, equatorial, and horizontal products where those products are in the claim
- exact named-exception behavior for out-of-coverage and disallowed-mode cases

The forbidden outcome is silent semantic drift.

---

## 10. Oracle-Hostile Requirement

The finished engine must also survive a higher-order audit:

- not merely "did it stay finite"
- but "did it remain semantically true against external authority"

For admitted products, the repository should maintain native-path oracle-hostile slices that test whether the public native route:

- remains aligned with external apparent references where those references govern the current product
- preserves event semantics rather than merely approximate event times
- preserves public chart-facing meaning rather than only low-level vector similarity

If the repository cannot test a claimed native-closed product against any higher authority, the finish claim should be narrowed accordingly.

---

## 11. Route-Law Requirement

Finished means the routing law is explicit.

The repository must be able to answer all of the following cleanly:

- which exact public products are native-routed
- which exact mode combinations are native-routed
- which exact products still fall back to Python
- whether any fallback is intentional, temporary, or disqualifying for the finish claim

No full-engine claim is honest if normal `.venv` execution still depends on opaque, incidental, or script-only route activation.

Production-route proof must occur in ordinary repository execution.

---

## 12. Benchmark Requirement

The native planetary engine is not finished merely because native code exists.

For the admitted closed products, benchmark artifacts must show:

- real public-workload measurement
- stable methodology
- honest baseline comparison
- material positive value or an explicitly accepted non-performance reason for closure

At minimum, the benchmark story must cover:

- `planet_at(...)`
- `all_planets_at(...)`
- any other product admitted into the closure claim

The repository must not call the engine finished if:

- reader-local wins exist
- but canonical public workloads remain effectively flat or negative

That would prove substrate closure, not engine closure.

---

## 13. Documentation Requirement

The finish claim is not complete until the architecture documents say the same thing the executable path says.

Before claiming completion, the repository should be able to update the architecture set so that:

- the path map no longer describes the closed products as Python-owned above the reader boundary
- the closure tracker rows for the claimed products are `Closed`
- any intentionally Python-retained rows are called intentional plainly
- the benchmark story is consistent with the claim
- the validation story names the exact parity, oracle, and adversarial surfaces supporting the claim

If the documents still need evasive wording to avoid overstating native ownership, the engine is not yet finished.

---

## 14. Concrete Exit Gates

The repository may truthfully say that the entire native planetary engine is finished only if all of the following gates are closed.

### Gate 1: Scope gate

The claim names the exact public products included and excluded.

### Gate 2: Boundary gate

The claimed products no longer rely on Python-owned core planetary computation above the declared native boundary.

### Gate 3: Python-parity gate

Native matches current Python public doctrine on the declared corpus with only execution-order residuals.

### Gate 4: External-validation gate

Existing external validation slices remain green with unchanged thresholds.

### Gate 5: Adversarial gate

Native public products survive seam, boundary, continuity, and named-error hostile tests.

### Gate 6: Oracle-hostile gate

Native public products remain semantically aligned with higher authority where such authority governs the product.

### Gate 7: Route-law gate

Normal `.venv` execution reaches the claimed native routes explicitly and repeatably.

### Gate 8: Benchmark gate

The public workload story is honestly positive or the closure claim is explicitly sovereignty-driven rather than speed-driven.

### Gate 9: Documentation gate

Architecture and validation documents all tell the same post-closure truth.

If any one gate is still open, the whole-engine finish claim remains premature.

---

## 15. Short Final Reading

The native DAF reader can already be treated as a validated substrate.

That is not the same thing as a finished native planetary engine.

The engine is finished only when the canonical public planetary products are:

- natively executed in their core computation
- parity-proven to current Python doctrine
- oracle-proven at the public product layer
- adversarially proven at the public product layer
- benchmark-proven on real workloads
- and documented without ambiguity

Until then, the truthful language is:

- the native planetary substrate is real
- the native planetary engine is still in closure
